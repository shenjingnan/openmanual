## 上下文

- openmanual 配置: !`cat docs/openmanual.json`
- 默认语言文件列表: !`find docs/content -maxdepth 1 -type d -name "*" | sed 's|.*/||' | grep -v content`
- 默认语言目录结构: !`find docs/content/zh -type f \( -name "*.mdx" -o -name "meta.json" \) 2>/dev/null | sort`

## 参数

- **目标语言代码** (`$ARGUMENTS`)：目标语言的 ISO 639-1 代码，如 `ko`、`ja`、`fr`、`de` 等

## 你的任务

将 OpenManual 文档站点的默认语言内容翻译为用户指定的目标语言。按照以下流程执行：

---

### 阶段 1：前置检查与场景判定

1. **读取配置**：解析 `docs/openmanual.json`，提取：
   - `i18n.enabled` 是否为 `true`
   - `i18n.defaultLanguage`（默认语言代码，通常为 `zh`）
   - `i18n.parser`（解析模式）
   - `i18n.languages` 数组

2. **场景判定**：

   **场景 A -- i18n 未开启**：如果 `i18n.enabled` 不为 `true` 或 `i18n` 字段不存在，输出提示并终止：
   > 当前项目未开启多语言功能（i18n.enabled 不为 true）。请先在 `docs/openmanual.json` 中设置 `"i18n": { "enabled": true, ... }` 再使用此命令。

   **场景 B -- 目标语言已存在（增量翻译）**：如果 `docs/content/<目标语言>/` 目录已存在，进入增量翻译模式。

   **场景 C -- 目标语言不存在（全新翻译）**：如果 `docs/content/<目标语言>/` 目录不存在，进入全新翻译模式。

3. **参数校验**：
   - 如果 `$ARGUMENTS` 为空或无效（非 2-3 位字母），提示用户输入正确的语言代码，例如 `/i18n-translate ko`
   - 如果目标语言与默认语言相同，提示无需翻译自身，终止执行

4. **确定源语言目录**：使用 `i18n.defaultLanguage` 的值作为源语言目录名，即 `docs/content/<defaultLanguage>/`

---

### 阶段 2：确定待翻译文件清单

#### 场景 B（增量翻译）-- 差异对比

对比源语言目录和目标语言目录：

1. **新增文件**：存在于源语言但不存在于目标语言的 `.mdx` 和 `meta.json` 文件
2. **内容变更文件**：两个目录中都存在但内容不同的文件（对每个共有文件用 diff 比较）
3. 输出差异报告给用户，列出新增和变更的文件数量及路径

#### 场景 C（全新翻译）-- 全量扫描

列出源语言目录下所有需要翻译的文件：
- 所有 `*.mdx` 文件
- 所有 `meta.json` 文件

输出全量文件清单给用户。

---

### 阶段 3：执行翻译

对阶段 2 确定的每个待翻译文件，按类型分别处理：

#### 3.1 翻译 meta.json 文件

1. 读取源语言的 `meta.json` 内容
2. 仅翻译 `title` 字段的值为目标语言
3. 保持 JSON 格式不变（2 空格缩进）
4. 写入目标路径

示例：
```json
// 源（中文）
{ "title": "指南" }

// 目标（韩文）
{ "title": "가이드" }
```

#### 3.2 翻译 MDX 文件

**Step 1 -- 读取源文件完整内容**

**Step 2 -- 翻译 frontmatter**
- 翻译 `title` 和 `description` 字段
- **保留所有其他字段不变**
- 保持 frontmatter 的 `---` 分隔符和格式不变

**Step 3 -- 翻译正文内容**

严格遵循以下保留规则：

| 元素 | 处理方式 |
|------|----------|
| 代码块（fenced code block） | **完全不翻译**，原样保留 |
| 行内代码（\`code\`） | 不翻译代码内容，可翻译周围说明文字 |
| MDX 组件标签（\<Component\>） | 不翻译组件名和属性名 |
| MDX 组件内的文本内容 | **翻译**组件标签之间的文本 |
| Mermaid 图表代码 | 不翻译图表代码，可翻译节点文本标签 |
| 表格 | 翻译单元格文本，保持表格结构 |
| 链接文本 | 翻译显示文本，**不改变** URL 路径 |
| 图片 alt 文本 | 翻译 alt 文本 |
| 标题（# ## ###） | 翻译标题文字 |
| 列表项 | 翻译列表项文字 |
| 强调/粗体/斜体 | 保留格式符号，翻译内部文字 |
| 引用块 | 翻译引用内容 |
| HTML 注释 | 不翻译 |

**Step 4 -- 输出翻译结果**
- 将翻译后的完整内容写入目标路径
- 保持 UTF-8 编码和原始换行风格

#### 3.3 翻译质量要求

- 技术文档风格：正式、准确、简洁
- 术语一致性：同一技术术语全文翻译一致
- 保留所有格式标记：MDX/Markdown 语法符号不得丢失或错位
- 代码块零改动：代码块内容必须逐字符一致

---

### 阶段 4：创建目录结构（仅场景 C）

对于全新翻译，在翻译文件之前：

1. 基于源语言目录结构，在 `docs/content/<目标语言>/` 下创建相同的子目录树
2. 先创建所有中间目录，再写入文件
3. 不复制文件本身（文件内容由翻译生成）

---

### 阶段 5：配置分析与更新

翻译完成后，分析 `openmanual.json` 中可能需要配合新语言更新的配置项：

#### 5.1 自动更新（无需确认）

**更新 `i18n.languages`**：
- 检查 `i18n.languages` 数组中是否已存在目标语言代码
- 如不存在，追加新条目：
  ```json
  { "code": "<目标语言>", "name": "<该语言的本地名称>" }
  ```
- 常见语言名称参考：
  - ko → 한국어, ja → 日本語, fr → Français, de → Deutsch
  - es → Español, pt → Português, ru → Русский, ar → العربية
- **不要移除或修改已有的语言条目**

#### 5.2 询问用户是否需要额外更新

翻译完成后，检查以下配置项并向用户询问是否需要一并处理：

**A. sidebar 标题翻译**

openmanual.json 的 `sidebar` 是统一配置（所有语言共享），其中包含默认语言的文本：
- `group` 字段：分组名称（如 "开始"、"指南"、"组件"、"进阶"、"社区"）
- `pages[].title` 字段：页面标题（如 "项目介绍"、"快速上手"、"配置参考" 等）

> **重要说明**：在 fumadocs dir-parser 模式下，侧边栏的实际显示标题优先取自各语言 MDX 文件的 frontmatter `title` 和对应目录的 `meta.json`（这两项你已在阶段 3 中翻译完毕）。sidebar 配置中的 title 主要作为 fallback 或覆盖值。如果你的站点依赖 sidebar config 中的 title 作为显示文本，则建议翻译；否则可以跳过。

**B. 其他可翻译字段**（如存在）

| 字段 | 说明 | 是否建议翻译 |
|------|------|-------------|
| `description` | 站点描述（SEO 用） | 建议翻译 |
| `footer.text` | 页脚文本 | 建议翻译 |

**执行方式**：

使用 `AskUserQuestion` 工具向用户提问，提供以下选项：

- **选项 1 — 全部处理**：翻译 sidebar 中所有 `group` 和 `pages[].title`，以及 `description`、`footer.text` 等其他可翻译字段
- **选项 2 — 仅翻译 sidebar**：只翻译 sidebar 中的分组名和页面标题
- **选项 3 — 仅翻译其他字段**：只翻译 `description`、`footer.text` 等，不动 sidebar
- **选项 4 — 跳过**：不修改上述任何内容，仅保留 `i18n.languages` 的更新

根据用户选择执行对应的更新操作。

**sidebar 翻译规则**：
- 只翻译 `group` 和 `pages[].title` 的字符串值
- **不修改** `slug`、`icon`、`collapsed` 等技术字段
- 保持 JSON 结构和缩进格式不变
- 翻译风格与 MDX 文档保持一致（同一术语翻译一致）

#### 5.3 提示后续步骤

> 翻译完成！已更新 openmanual.json 的相关配置。
>
> 后续步骤：
> 1. 运行 `pnpm run docs:build` 验证构建是否成功
> 2. 运行 `pnpm run docs:dev` 本地预览翻译效果
> 3. 确认无误后提交代码

---

### 阶段 6：输出总结

翻译全部完成后，输出总结：

```
## 翻译完成总结

- **目标语言**: <语言代码> (<语言名称>)
- **翻译模式**: 全新翻译 / 增量翻译
- **处理文件数**: <N> 个
  - MDX 文件: <N> 个
  - meta.json: <N> 个
- **跳过文件数**: <N> 个（仅增量模式，无需更新的文件）
- **配置更新**:
  - i18n.languages: 已追加 <语言代码> 条目
  - sidebar: <已翻译 / 已跳过>
  - 其他字段: <已翻译 / 已跳过 / 无需更新>
- **输出目录**: docs/content/<目标语言>/
```

---

### 边界情况处理

1. **目标语言与默认语言相同**：提示用户无需翻译自身，终止执行
2. **源语言目录为空**：提示用户源语言目录没有可翻译的内容，终止执行
3. **MDX 文件无 frontmatter**：正常翻译正文，不强制添加 frontmatter
4. **meta.json 含有除 title 外的其他字段**：只翻译 title，保留其他字段不变
5. **父目录不存在**：先创建父目录再写入文件
6. **目标语言代码已在 languages 中但目录不存在**：按场景 C 处理，不再重复追加 languages 条目
7. **超大文件**（>10KB）：分块翻译，确保不截断内容
8. **特殊字符**：正确处理 MDX 中的 `{}`、`<>`、`\` 等特殊字符，不得破坏 JSX 表达式
9. **翻译内容为空**：不写入文件，记录警告并继续下一个
